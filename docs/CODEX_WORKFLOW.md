# CODEX WORKFLOW — UFO Predictor

_Last updated: post C05 / pre C06_

Current baseline: `main` is post PR #29 (`Feature/registered free saved matches`). C05 is functionally closed. Next major block: C06 — World Cup Premium Package Foundation.


## Purpose

This document defines how ChatGPT, Codex, manual PowerShell/Git, Supabase SQL Editor, and GitHub should be used in the UFO Predictor project.

## Role Split

### ChatGPT

Use ChatGPT for:

- product direction;
- technical planning;
- scope control;
- reviewing Codex responses;
- deciding implementation boundaries;
- writing clean Codex prompts;
- documentation refreshes at stage close/handoff.

### Codex

Use Codex for:

- repository inspection when reasoning across multiple files is needed;
- implementation;
- refactors;
- SQL/migration file creation;
- file-level technical reports tied to a task;
- validation reports when it is already working on the repo.

Do not use Codex just to run trivial terminal commands.

### Manual PowerShell/Git

The user should run simple commands manually and paste results when needed:

- `git status`
- `git status --short`
- `git diff`
- `git diff --stat`
- `git log --oneline -5`
- `git branch`
- `git add`
- `git commit`
- `git push`
- `npm run lint`
- `npm run test`
- `npm run build`
- targeted `rg` / `Select-String` checks

This avoids spending Codex capacity on work the user can perform faster.

## PR / Merge Discipline

Do not merge every micro-step to `main`.

Use a longer feature branch with multiple commits when a block has one coherent functional goal.

Merge to `main` when:

- the feature has functional value;
- a security/data boundary is actually closed;
- a stage is complete;
- the branch is validated and ready for review.

Do not merge to `main` for:

- recognition;
- planning;
- partial implementation without usable value;
- docs notes after every small change;
- micro-steps that are better kept as commits inside one branch.

## Documentation Refresh Discipline

Do not update docs after every small progress step.

Update docs when:

- closing an epic/stage;
- switching conversation;
- changing major product or architecture decisions;
- preparing Codex/ChatGPT handoff.

Documentation refreshes should be conservative:

- preserve existing context;
- avoid destructive summarization;
- add new baseline and decisions;
- remove or correct obsolete contradictions;
- avoid rewriting broad docs into short summaries.

## Supabase Workflow

Supabase CLI local is not the normal workflow.

Codex may create migration files, but the user applies SQL manually in Supabase SQL Editor.

For migrations, Codex must provide:

1. SQL file path.
2. Complete SQL.
3. Manual validation queries.
4. Explicit statement that remote has not been applied.

The user then:

1. Applies SQL manually in Supabase SQL Editor.
2. Runs validation queries.
3. Shares results with ChatGPT.
4. Runs local validation.
5. Commits only after remote validation passes.

## Security Rules For Codex

- Visual locks/blur/teasers are not authorization.
- Do not send premium payload to the browser without server-side authorization.
- Do not use service role for normal UI.
- Do not open `prediction_markets`, `prediction_narratives`, or `prediction_results` unless explicitly approved.
- Do not treat `premium_user` role alone as authorization.
- Do not treat active subscription alone as authorization.
- Do not treat `quantity/match_pack` as direct authorization.
- `trustedBetaFreeMatchIds` must come from trusted server-side context.
- `stageAccessKey` must be canonical and server-derived.

- After that, return to feature branch -> PR -> merge.

- C05 saved matches correctly used multiple commits in one branch before PR #29.

## Documentation Refresh Timing

Do not refresh project documentation after every small implementation step. Refresh docs when one of these is true:

- a stage/epic closes;
- a new conversation or major handoff is about to start;
- an important product/architecture/security decision changed;
- a migration or access boundary change must become the new baseline.

For long-running feature work, keep intermediate decisions in the active conversation. Commit code in the feature branch as needed, but defer broad documentation refresh until the block is ready for handoff.
